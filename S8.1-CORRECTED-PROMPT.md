## TASK 1: PROACTIVE NOTIFICATION PREFERENCE (replaces "marketing consent")

### Why the rename
These templates are Utility on Meta, not Marketing. Calling the column
`whatsapp_marketing_consent` is misleading. Use `whatsapp_proactive_enabled`
instead — it accurately describes what it controls: whether the org receives
proactive scheduled notifications (morning summary, payment reminders,
compliance alerts).

### 1a. Database migration

Create `supabase/migrations/YYYYMMDDHHMMSS_add_whatsapp_proactive_preference.sql`:

```sql
-- Add proactive notification preference to organizations
ALTER TABLE organizations
  ADD COLUMN whatsapp_proactive_enabled  BOOLEAN NOT NULL DEFAULT TRUE,
  ADD COLUMN whatsapp_proactive_set_at   TIMESTAMPTZ;

-- Backfill: existing orgs get TRUE (opted in by default)
-- whatsapp_proactive_set_at stays NULL for existing orgs (implies "default, never explicitly set")

COMMENT ON COLUMN organizations.whatsapp_proactive_enabled IS
  'Whether this org receives proactive WhatsApp notifications (morning summary, payment reminders, compliance alerts). UX preference, not a Meta marketing requirement — all templates are Utility category.';
COMMENT ON COLUMN organizations.whatsapp_proactive_set_at IS
  'When the owner last changed the proactive notification preference. NULL = never explicitly set (using default).';
```

After creating the migration, regenerate types:
```bash
npx supabase gen types ts --local > src/types/database.ts
```

### 1b. Update SCHEMA.md

Add the two new columns to the organizations table definition in

whatsapp_proactive_enabled  BOOLEAN NOT NULL DEFAULT TRUE  -- receives scheduled notifications (summary, reminders, compliance)
whatsapp_proactive_set_at   TIMESTAMPTZ                    -- when the preference was last explicitly changed

### 2c. Payment reminder workflow — add consent check

**File:** `n8n/workflows/payment-reminder.json`

The payment reminder workflow sends to **customers** (not owners), so the
proactive preference toggle doesn't directly apply — owners opt out of
messages sent to themselves, not of reminders sent to their customers.

notifications, should their customers also stop getting payment reminders?

**Decision: NO.** Payment reminders are a core business function tied to
overdue invoices. The proactive toggle controls messages sent TO the owner.
Customer-facing payment reminders continue regardless. This is consistent
with the tiered payment reminders being Utility category on Meta.

No changes needed to payment-reminder.json for consent. It already works
correctly with template sends.

### 3c. Do NOT add /orders as a slash command

The flow-engine.ts slash command handler (line 683) supports:
- `/status` — order summary for a specific customer chat
- `/cancel` — cancel active pending order
- `/edit <text>` — re-run extraction with new text
- `/order <text>` — manual order trigger

`/orders` (plural, "show me all my orders") is NOT needed because:
- The owner has the web dashboard for that
- Daily morning summary already covers the aggregate view
- Adding it would violate the principle of minimal bot surface area

**No code change.** Just ensure the template and API don't reference it.
